# Сервис сокращения ссылок (Link Shortener)

Готовый к производству сервис сокращения ссылок с управлением, аналитикой и организацией по проектам, построенный на FastAPI, PostgreSQL, Redis. 

## Оффтоп

Сервис совсем базовый по сути - сложной/специфической логики тут нет, поэтому было принято стратегическое решение описать все кратенько в ридмишке и не заниматься спавном миллиарда докстрингов под каждую функцию. Аля '''Функция создания пользователя''' (да и честно по ДДД все так расползлось по кол-ву файлов что я уже и не вывезу это написание комментов, особенно после правки пары моментов из-за которых пришлось бежать по всей структуре от моделей к юзкейсам  и менять код (и надеяться что ничего из-за обратной несовместимости в соседних юзкейсах не сломается))

## Возможности

- **Сокращение URL**: Генерация коротких кодов для URL
- **Пользовательские алиасы**: Можно поставить свой короткий код
- **TTL / Истечение срока**: Ссылки могут иметь дату протухания (с автоматической очисткой в фоне)
- **Статистика ссылок**: Отслеживание кликов, временных меток последнего использования и графиков использования
- **Кэширование Redis**: Кэш для быстрого поиска редиректов
- **Ограничение запросов**: Ограничение запросов для API-эндпоинтов (по IP)
- **Аутентификация**: Регистрация пользователей и вход на основе JWT
- **Управление ссылками**: Полный CRUD для ссылок (создание, чтение, обновление, удаление, поиск (тут мягкое удаление (флагом)))
- **Поддержка проектов**: Организация ссылок по проектам (тоже круд, удаление проектов жесткое)
- **Демо-интерфейс**: Статика отдаваемая с бека чтобы потфкать удобнее чем в свагере (сначала подумал раз в требованиях сервис то и фронт нужен, решил оставить)
- **Нагрузочное тестирование с Locust**: Сейчас тестирование упирается в рейт лимиты (они отрабатывают и в принципе их можно крутить по желанию, работает нормально)
- **Докер**: Весь проект под запуск вместе с PostgreSQL и Redis

## Архитектура

Сервис следует принципам **Чистой архитектуры (почти, я пытался) + DDD**:

- **Слой представления (Presentation Layer)**: FastAPI роутеры, зависимости, Pydantic-схемы, статический фронтенд
- **Слой приложения (Application Layer)**: Сценарии использования (use cases), DTO, сервисы (ShortCodeGenerator, TokenProvider и т.д.)
- **Доменный слой (Domain Layer)**: Сущности (Link, User, Project, LinkStats), объекты-значения (ShortCode, OriginalUrl, ExpiresAt), интерфейсы репозиториев
- **Инфраструктурный слой (Infrastructure Layer)**: База данных (SQLAlchemy модели, мапперы, единица работы), кэш Redis, JWT-аутентификация, фоновые задачи (APScheduler)

## Технологический стек

- **Python 3.12+**
- **FastAPI** — Современный веб-фреймворк с автоматической документацией OpenAPI
- **SQLAlchemy 2.x (async)** — ORM с асинхронной поддержкой
- **PostgreSQL** — Основная база данных
- **Redis** — Кэширование и ограничение запросов
- **Alembic** — Миграции базы данных
- **JWT** — Аутентификация с python-jose
- **APScheduler** — Планирование фоновых задач
- **Pydantic & Pydantic Settings** — Валидация данных и управление конфигурацией
- **Uvicorn** — ASGI-сервер
- **Locust** — Фреймворк для нагрузочного тестирования

## Переменные окружения

Сервис настраивается через переменные окружения. Скопируйте `.env.example` в `.env` и настройте по необходимости.

| Переменная | Описание | Пример | Обязательно |
|------------|----------|--------|-------------|
| `DATABASE_URL` | Строка подключения к PostgreSQL | `postgresql+asyncpg://user:password@localhost:5432/link_shortener` | Да |
| `REDIS_URL` | URL подключения к Redis | `redis://localhost:6379/0` | Да |
| `JWT_SECRET` | Секретный ключ для подписи JWT-токенов | `your-secret-key-change-in-production` | Да |
| `JWT_ALGORITHM` | Алгоритм подписи JWT | `HS256` | Нет (по умолчанию: HS256) |
| `JWT_EXPIRES_MIN` | Время жизни JWT-токена в минутах | `30` | Нет (по умолчанию: 30) |
| `BASE_URL` | Базовый URL сервиса | `http://localhost:8000` | Нет (по умолчанию: http://localhost:8000) |
| `SHORT_LINK_PREFIX` | Префикс пути для редиректов | `opupupa` | Нет (по умолчанию: opupupa) |
| `DEFAULT_CACHE_TTL_SEC` | TTL кэша Redis в секундах | `86400` (24 часа) | Нет (по умолчанию: 86400) |
| `PURGE_INTERVAL_SEC` | Интервал для задачи очистки просроченных ссылок | `300` (5 минут) | Нет (по умолчанию: 300) |
| `UNUSED_LINK_TTL_DAYS` | Дней, после которых неиспользуемые ссылки считаются устаревшими | `90` | Нет (по умолчанию: 90) |
| `DEBUG` | Включить режим отладки | `true` | Нет (по умолчанию: false) |
| `LOG_LEVEL` | Уровень логирования | `INFO` | Нет (по умолчанию: INFO) |
| `CORS_ORIGINS` | Разделенные запятыми источники CORS | `http://localhost:3000,http://localhost:8000` | Нет (опционально) |

## Запуск с Docker

Проект включает полную настройку Docker Compose с PostgreSQL, Redis и приложением.

### Быстрый старт

1. **Сборка и запуск:**
   ```bash
   docker-compose up --build
   ```

2. **Доступ к приложению:**
   - Приложение: http://localhost:8000
   - PostgreSQL: localhost:5432
   - Redis: localhost:6379

### Сервисы

- **db**: PostgreSQL 16 с проверками здоровья и постоянным хранилищем
- **redis**: Redis 7-alpine с постоянным хранением через append-only файл
- **app**: Приложение на Python 3.12-slim с автоматическими миграциями при запуске

### Детали Docker Compose

`docker-compose.yml` включает:
- Автоматические миграции базы данных при запуске приложения
- Healthcheck для базы данных и Redis
- Постоянное хранилище для данных PostgreSQL и Redis

## Нагрузочное тестирование

### Запуск нагрузочных тестов

1. **Запуск приложения** 

2. **Запуск Locust:**
   ```bash
   locust -f tests/load/locustfile.py --host=http://localhost:8000
   ```

3. **Веб-интерфейс Locust** по адресу http://localhost:8089

### Сценарии нагрузочного тестирования

Файл locustfile включает три класса пользователей:

1. **LinkCreationUser** (вес: 3)
   - Создает короткие ссылки со случайными пользовательскими алиасами
   - Создает короткие ссылки без пользовательских алиасов (генерируемые системой)
   - Обрабатывает ограничение запросов и коллизии алиасов корректно

2. **RedirectUser** (вес: 10)
   - Моделирует высокообъемный трафик редиректов с использованием предварительно заполненного пула коротких кодов
   - Иногда создает новые ссылки (низкая частота)

3. **CacheEfficiencyUser** (вес: 8)
   - Тестирует производительность кэша, сравнивая сценарии холодного и теплого кэша
   - Создает высокую нагрузку для стресс-тестирования кэширующего слоя

### Подготовка тестовых данных

Перед началом теста Locust автоматически создает 100 начальных коротких кодов (пулл ссылок для тестирования редиректов, чтобы не долбить нереалистичный рейт создания).
